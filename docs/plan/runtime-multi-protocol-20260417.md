# Runtime 多协议适配器设计

> **日期**: 2026-04-17
> **状态**: 草案
> **作者**: Zoumo + Claude

## 1. 背景与动机

### 现状

MASS runtime 与 ACP（Agent Control Protocol）紧密耦合，耦合点遍布各层：

- `pkg/agentrun/runtime/acp/runtime.go` 是**唯一的** runtime 实现
- `Config.AcpAgent` 是**唯一的** agent 配置类型 — 字段名直接写死 "acp"
- `Manager.Events()` 返回 `<-chan acp.SessionNotification`（ACP SDK 类型）
- `Translator` 硬编码了 ACP 通知类型的 type-switch
- Bootstrap 序列硬编码了 `acp.NewClientSideConnection()` → `Initialize()` → `NewSession()`
- 权限模型硬编码了 ACP 的 `RequestPermission()` 回调

### 问题

我们需要支持多种真实 agent，每种都有不同的协议：

| Agent | 传输层 | 协议 | 备注 |
|-------|--------|------|------|
| ACP agents | stdio | ACP 协议 | 当前已支持 |
| Claude Code | stdio | JSON-RPC | System prompt 通过 CLI 参数注入，支持结构化输出 |
| Codex | stdio | 自定义 | 自有协议 |
| GSD | RPC | JSON-RPC | 自定义 RPC 模式 |
| Pi | RPC | JSON-RPC | 自定义 RPC 模式 |

每个 agent 协议各不相同，但大多数共享 **stdio + JSON-RPC** 作为通信基础。

### 目标

1. **解耦 runtime 与 ACP** — ACP 变为众多协议之一
2. **支持原生 agent CLI** — 直接 fork/exec，按协议通信
3. **保持 agent-run + RPC 接口不变** — 外部 API（ARI + shim RPC）不动
4. **原生 agent 能力 ≥ ACP** — 例如 CLI 注入 system prompt、结构化输出
5. **只设计终态** — 不考虑迁移，干净架构

### 非目标

- 迁移策略或向后兼容
- 新增 ARI 方法（ARI 保持不变）
- 新增 shim 二进制文件（agent-run shim 保持不变，内部适配）

## 2. 总体架构方向：方案 C — 基础 Runtime + 协议混入

经过三种方案对比（详见附录 A），选定方案 C。核心思路：

- `BaseRuntime` 处理通用逻辑（进程管理、状态机、事件分发）
- `Protocol` 接口处理协议特定逻辑（握手、消息编码、事件翻译）
- 新 agent 只需实现 Protocol，而非完整 Runtime

**当前阶段聚焦 API 层变更**，为未来内部重构铺路。

### 不变量

```
┌─────────────┐     ARI (JSON-RPC)     ┌──────────┐
│  调用方       │ ◄──────────────────── │  agentd   │
│  (massctl)   │                        │           │
└─────────────┘                        └─────┬────┘
                                              │ fork/exec
                                              ▼
                                       ┌──────────┐   shim RPC (JSON-RPC)
                                       │ agent-run │ ◄──── agentd 重连
                                       │  (shim)   │
                                       └─────┬────┘
                                              │ ← 这部分要改
                                              ▼
                                       ┌──────────┐
                                       │  Agent    │
                                       │  进程     │
                                       └──────────┘
```

**保持不变**：
- agentd ↔ agent-run：shim RPC（session/prompt、session/cancel、runtime/watch_event 等）
- agentd ↔ 调用方：ARI（agent/create、agentrun/prompt 等）
- 事件系统：AgentRunEvent + sealed Event 接口 + seq/turnId/streamSeq
- 状态机：creating → idle → running → stopped → error
- 状态持久化：state.json、events.jsonl、agent-run.sock
- Bundle 结构：config.json + workspace 软链接 + 状态文件

## 4. API 类型变更详细设计

### 4.1 核心设计原则

**协议只是通信方式，不影响配置结构。**

所有 agent 都需要：启动进程（Process）、会话配置（Session）。唯一的差异是 agent-run 与 agent 进程之间用什么协议通信。因此用一个 `ClientProtocol` 枚举字段区分，其余字段全部共享。

### 4.2 Config 变更（`pkg/runtime-spec/api/config.go`）

**当前**：

```go
type Config struct {
    MassVersion string           `json:"massVersion"`
    Metadata    Metadata         `json:"metadata"`
    AgentRoot   AgentRoot        `json:"agentRoot"`
    AcpAgent    AcpAgent         `json:"acpAgent"`        // ← ACP 专属
    Permissions PermissionPolicy `json:"permissions,omitempty"`
}

type AcpAgent struct { ... }   // ← ACP 专属
type AcpProcess struct { ... } // ← ACP 专属
type AcpSession struct { ... } // ← ACP 专属
```

**改为**：

```go
// Config is the top-level MASS Runtime Specification config.json structure.
// Generated by agentd, consumed by agent-run.
type Config struct {
    // MassVersion is the MASS Runtime Spec version (SemVer).
    MassVersion string `json:"massVersion"`

    // Metadata describes the identity of this agent instance.
    Metadata Metadata `json:"metadata"`

    // AgentRoot specifies the agent's root working directory within the bundle.
    AgentRoot AgentRoot `json:"agentRoot"`

    // ClientProtocol specifies the communication protocol between agent-run
    // and the agent process. Determines how bootstrap, prompt, and event
    // streaming are handled. Default: ClientProtocolACP.
    ClientProtocol ClientProtocol `json:"clientProtocol,omitempty"`

    // Process specifies how to launch the agent process (fork/exec params).
    Process Process `json:"process"`

    // Session specifies session-level configuration: system prompt,
    // permissions, and MCP servers.
    Session Session `json:"session,omitempty"`
}

// ClientProtocol specifies the communication protocol between agent-run
// and the agent process.
type ClientProtocol string

const (
    // ClientProtocolACP communicates via ACP protocol over stdio.
    ClientProtocolACP ClientProtocol = "acp"

    // ClientProtocolClaudeCode communicates via Claude Code's native
    // JSON-RPC protocol over stdio.
    ClientProtocolClaudeCode ClientProtocol = "claude-code"

    // ClientProtocolCodex communicates via Codex's native protocol over stdio.
    ClientProtocolCodex ClientProtocol = "codex"
)

// IsValid reports whether p is a known ClientProtocol value.
func (p ClientProtocol) IsValid() bool {
    switch p {
    case ClientProtocolACP, ClientProtocolClaudeCode, ClientProtocolCodex:
        return true
    }
    return false
}

// Process specifies how to launch an agent process.
// Shared across all client protocols.
type Process struct {
    // Command is the agent executable.
    Command string `json:"command"`

    // Args are command-line arguments.
    Args []string `json:"args,omitempty"`

    // Env is environment variable overrides in KEY=VALUE format.
    // agent-run merges parent process env as base, these take precedence.
    Env []string `json:"env,omitempty"`
}

// Session specifies session-level configuration.
// Shared across all client protocols.
type Session struct {
    // SystemPrompt is the agent's role definition and capability constraints.
    // How it is delivered depends on ClientProtocol:
    //   - ACP: via session/new or first prompt (ACP bootstrap)
    //   - claude-code: via CLI args
    //   - Others: protocol-specific
    SystemPrompt string `json:"systemPrompt,omitempty"`

    // Permissions controls how agent-run handles agent-initiated
    // fs/* and terminal/* requests. Default: ApproveAll.
    Permissions PermissionPolicy `json:"permissions,omitempty"`

    // McpServers is the list of MCP services available to the agent.
    McpServers []McpServer `json:"mcpServers,omitempty"`
}
```

**删除的类型**：`AcpAgent`、`AcpProcess`、`AcpSession` — 全部废弃。

### 4.4 AgentSpec 变更（`pkg/ari/api/domain.go`）

**当前**：

```go
type AgentSpec struct {
    Command               string   `json:"command"`
    Args                  []string `json:"args,omitempty"`
    Env                   []EnvVar `json:"env,omitempty"`
    StartupTimeoutSeconds *int     `json:"startupTimeoutSeconds,omitempty"`
}
```

**改为**：

```go
type AgentSpec struct {
    // ClientProtocol selects the communication protocol adapter.
    // Default: "acp".
    ClientProtocol apiruntime.ClientProtocol `json:"clientProtocol,omitempty"`

    // Command is the agent executable.
    Command string `json:"command"`

    // Args are command-line arguments.
    Args []string `json:"args,omitempty"`

    // Env is environment variable overrides.
    Env []EnvVar `json:"env,omitempty"`

    // StartupTimeoutSeconds is the bootstrap timeout.
    StartupTimeoutSeconds *int `json:"startupTimeoutSeconds,omitempty"`
}
```

变更极小：只加一个 `ClientProtocol` 字段。agentd 在生成 config.json 时将此值传递到 `Config.ClientProtocol`。

### 4.5 ValidateConfig 变更（`pkg/runtime-spec/config.go`）

```go
func ValidateConfig(c api.Config) error {
    // 1. Validate massVersion (unchanged)
    // 2. Validate metadata (unchanged)
    // 3. Validate agentRoot (unchanged)

    // 4. Validate clientProtocol
    protocol := c.ClientProtocol
    if protocol == "" {
        protocol = api.ClientProtocolACP // default
    }
    if !protocol.IsValid() {
        return fmt.Errorf("unknown clientProtocol: %q", protocol)
    }

    // 5. Validate process
    if c.Process.Command == "" {
        return fmt.Errorf("process.command must not be empty")
    }

    // 6. Validate session.permissions (if set)
    if c.Session.Permissions != "" && !c.Session.Permissions.IsValid() {
        return fmt.Errorf("invalid session.permissions: %q", c.Session.Permissions)
    }

    return nil
}
```

### 4.6 agentd Config 生成

agentd 从 `AgentSpec` + `AgentRunSpec` 生成 `Config`：

```go
func buildConfig(agent Agent, run AgentRunSpec) api.Config {
    protocol := agent.Spec.ClientProtocol
    if protocol == "" {
        protocol = api.ClientProtocolACP
    }

    return api.Config{
        MassVersion:    "0.1.0",
        Meta       api.Metadata{Name: run.Name},
        AgentRoot:      api.AgentRoot{Path: "workspace"},
        ClientProtocol: protocol,
        Process: api.Process{
            Command: agent.Spec.Command,
            Args:    agent.Spec.Args,
            Env:     resolveEnv(agent.Spec.Env),
        },
        Session: api.Session{
            SystemPrompt: run.SystemPrompt,
            Permissions:  api.ApproveAll,
            McpServers:   resolveMcpServers(run.McpServers),
        },
    }
}
```

所有协议共用一套生成逻辑。

## 6. 关键设计决策

| # | 决策 | 理由 |
|---|------|------|
| D-R01 | `ClientProtocol` 枚举替代 ACP 硬编码 | 一个字段区分协议，其他全部共享，零冗余 |
| D-R02 | `ClientProtocol` 默认 `"acp"` | 向后兼容，省略时行为不变 |
| D-R03 | `Process` 所有协议共享 | 所有 agent 都需要 fork/exec，进程配置与协议无关 |
| D-R04 | `Session` 统一包含 systemPrompt + permissions + mcpServers | 都是会话级配置，投递方式由 clientProtocol 决定 |
| D-R05 | `Permissions` 从 Config 顶层移入 Session | 属于会话级语义，与 systemPrompt 和 mcpServers 同层 |
| D-R06 | `AcpAgent`/`AcpProcess`/`AcpSession` 全部废弃 | 无需保留 ACP 专属类型，所有字段已在共享类型中覆盖 |
| D-R07 | `AgentSpec` 只加一个 `ClientProtocol` 字段 | ARI 层面的协议选择，最小改动 |
| D-R08 | 不加 `protocolConfig` 扩展字段 | YAGNI — 等遇到真实需求再加 |
| D-R09 | session.go/state.go 不变 | 已是协议无关（AgentInfo、Capabilities 等） |

## 7. 待定问题

1. **McpServer 是否所有协议都支持**：ACP 原生支持 MCP。Claude Code 也支持 MCP（通过 CLI 参数）。但不是每个 agent 都支持 MCP。不支持的 agent 忽略 `session.mcpServers` 即可，无需在 API 层限制。
2. **Permissions 是否所有协议都支持**：ACP 有明确的 permission 回调。其他协议可能没有 permission 概念。不支持的协议忽略此字段即可。
3. **新协议的添加流程**：在 `ClientProtocol` 枚举中添加新常量 + `IsValid()` 更新 + 内部实现新 Protocol 适配器。API 层改动极小。

## 附录 A：方案对比

### 方案 A：策略模式 — Runtime 接口 + 完整适配器

**优点**：简单直观、自包含、易测试、职责清晰
**缺点**：代码重复约 60%（进程管理、状态机）、一致性风险、维护负担

### 方案 B：分层协议栈 — Transport × Protocol × Capability

**优点**：最大灵活性、无重复、可组合
**缺点**：过度工程化、组合爆炸、调试困难、抽象泄漏、真实 agent 协议无法干净分层

### 方案 C：基础 Runtime + 协议混入（选定）

**优点**：无重复、Protocol 接口聚焦（约 300 行）、不变量集中强制、两级测试
**缺点**：BaseRuntime 复杂度需控制、Protocol 接口设计关键

| 维度 | A | B | C（选定） |
|------|---|---|----------|
| 复杂度 | 低 | 高 | 中 |
| 代码重复 | 约 60% | 无 | 无 |
| 新增 agent | 约 800 行 | 仅配置（理想）| 约 300 行 |
| 适合真实 agent | 是 | 差 | 是 |

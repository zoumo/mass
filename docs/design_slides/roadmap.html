§ 当前状态分析

项目已拥有一个可工作的 agent-shim 层 — 这是 OAR Runtime Spec 的参考实现， 覆盖了架构栈的底层部分。 

Implemented:
  pkg/spec       — OAR Runtime Spec types, config parsing, state management
  pkg/runtime    — Manager: agent process lifecycle, ACP handshake, permissions
  pkg/events    — Typed event stream, EventLog (JSONL), ACP→Event translator
  pkg/rpc        — JSON-RPC 2.0 server over Unix socket (shim RPC)
  cmd/agent-shim — CLI entry point
  cmd/agent-shim-cli — Interactive client (state, prompt, chat, shutdown)

Not yet implemented:
  agentd         — High-level daemon (Workspace/Session/Process/Room Manager)
  Orchestrator   — Room lifecycle, multi-agent coordination

关键洞察： agent-shim 是整个架构的基石。任何在此层的 Bug 都会传播到上层所有模块， 因此 Phase 1 的首要任务是加固这一基础。

§ Phase 1: agent-shim 加固

在构建上层之前，首先巩固现有的 agent-shim 层。这一阶段包含四个核心任务组。

阶段进度 0%

1.1 Terminal Operations

状态：Stub 方法在 pkg/runtime/client.go 中返回 "not supported"。

实现 terminal/execute — 在 agent workspace 中执行 shell 命令
实现 terminal/read_output — 流式读取命令输出
实现 terminal/set_timeout — 配置执行超时
添加 terminal 操作的权限策略执行
集成测试: mockagent 发起 terminal 请求

1.2 Session Load (Warm Resume)

状态：ACP session/load 在 spec 中有引用但未接入。

在 Manager 中实现 session/load — 恢复对话历史
恢复停止的 agent 时，在 session/prompt 之前接入 load
持久化对话历史 (turns) 以支持冷重启

1.3 Graceful Shutdown & Error Propagation

将 agent 进程退出错误传播到 state.json (添加 exitCode, error 字段)
在 shim RPC GetState() 响应中暴露进程退出原因
处理 ACP 握手期间的 agent 进程崩溃 (清理部分状态)
添加可配置的 SIGTERM→SIGKILL 升级等待期 (当前硬编码 5s)

1.4 Event Log Improvements

添加事件日志轮转或大小限制 (当前无界追加)
支持 GetHistory 带时间范围和事件类型过滤
事件吞吐量基准测试，必要时优化

§ Phase 4: Room Manager

Room Manager 解锁多 agent 协作 — 这是 OAR 的差异化特性。Room 定义了 agent 间的通信拓扑。

阶段进度 0%

通信模式

mesh

任意 agent 可向任意其他 agent 发送消息

star

仅 leader 可发起; 其他 agent 仅能回复

isolated

无 agent 间通信

MCP Tool Injection

room_send(agent_name, message)  — Route to target session's prompt
room_broadcast(message)       — Fan-out to all room members
room_status()                 — Return member states

Busy session handling: 当目标 session 正在处理时返回错误

ARI Room Methods
room/create room/status room/delete

room/delete 要求所有成员 session 已停止

关键洞察： Phase 4 依赖 Phase 3 的 Workspace Manager，因为 Room 需要共享工作空间支持。 两者可在 Phase 2 完成后并行开发。

§ Phase 5: Session Lifecycle

实现 Warm/Cold Pause — 资源优化机制，在不牺牲功能的前提下减少运行开销。

阶段进度 0%

Warm Pause

  • 检测 turn 完成 → 转换到 Paused:Warm
  • 可配置 warm idle timeout (sessionPolicy.warmIdleTimeout)
  • Max warm sessions 限制 (sessionPolicy.maxWarmSessions)
  • Resume: 直接发送下一个 prompt

Agent 进程保持运行 — 快速恢复，无状态加载开销

Cold Pause

  • Warm idle timeout → kill agent process → Paused:Cold
  • 持久化对话历史用于 cold restart
  • Cold retention timeout (sessionPolicy.coldRetentionTimeout)
  • Resume: restart process → session/load → prompt

Agent 进程已终止 — 释放资源，恢复需重启 + 加载状态

关键洞察： Phase 5 是优化而非核心功能。有用但不阻塞主要用例。 可在 Phase 2-4 完成后实施。

§ Phase 6: Production Readiness

§ 优先级原理

每个阶段的优先级都有明确的工程理由支撑。

1

Phase 1 first

agent-shim 是基础 — 此层的 Bug 会传播到所有上层模块

2

Phase 2 highest value

agentd 使管理多个 agent 成为可能 — 这是核心用例

3

Phase 3 before Phase 4

Room 需要共享工作空间 — Workspace Manager 是前置依赖

4

Phase 4 differentiating feature

多 agent 协作是 OAR 的差异化特性

5

Phase 5 optimization

Warm/Cold Pause 是优化 — 有用但不阻塞核心功能

6

Phase 6 continuous

生产就绪持续进行，但关键项（日志、恢复）应早期启动

◈ 路线图总结

从底层 agent-shim 加固开始，逐步构建 agentd 核心能力， 然后并行开发 Workspace 和 Room Manager， 最后优化 session 生命周期并确保生产就绪。 每一步都有明确的依赖关系和价值驱动。

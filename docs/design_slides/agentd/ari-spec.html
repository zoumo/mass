§1 架构设计

什么是 ARI？

ARI (Agent Runtime Interface) 是 orchestrator 和 agentd 之间的接口协议。 它定义了如何创建 session、管理 workspace、与运行中的 agent 交互以及管理 room。

ARI 的设计参照了 Kubernetes 的 CRI (Container Runtime Interface)， 后者是 kubelet 和 containerd 之间的 gRPC 接口。CRI 定义了编排器如何请求容器操作，而不耦合到任何特定的运行时实现。 ARI 服务于相同目的——为 Agent 运行时提供标准化的操作接口。

CRI → ARI 映射

以下是 CRI 服务与 ARI 方法的对应关系：

CRI Service ARI 对应 说明
RuntimeService.RunPodSandboxroom/*创建协同调度组
RuntimeService.CreateContainersession/new创建工作负载实例
RuntimeService.StartContainer（隐含在 session/new 中）推迟分离
RuntimeService.StopContainersession/stop停止工作负载
RuntimeService.RemoveContainersession/remove清理元数据
RuntimeService.ListContainerssession/list查询工作负载
RuntimeService.ContainerStatussession/status查询工作负载状态
RuntimeService.ExecSyncsession/prompt在 session 中执行工作
ImageService.PullImageworkspace/prepare准备工作环境
ImageService.RemoveImageworkspace/cleanup移除工作环境
ImageService.ListImagesworkspace/list查询可用环境

CRI 有但 ARI 不需要

  • • ContainerStats / PodSandboxStats — Agent 不需要资源监控
  • • UpdateContainerResources — 不需要运行时资源限制
  • • PortForward — 没有 network namespace 可以转发
  • • Attach — ARI 有自己的 attach 语义用于交互式 session

ARI 有但 CRI 没有

  • • session/prompt — Agent 处理 prompt，这是核心的 agent 交互模型
  • • session/cancel — 取消进行中的 agent 工作（遵循 ACP 语义）
  • • session/attach / session/detach — 交互式观察 agent 工作
  • • room/send / room/broadcast — Room 内的 agent 间通信

§2 传输协议

📡

JSON-RPC 2.0 over Unix Socket

路径: /run/agentd/agentd.sock（可配置）

为什么使用 JSON-RPC over Unix Socket？

1 仅本地通信

不需要 TCP，Unix socket 专为本地进程间通信优化

2 内置访问控制

Unix socket 权限（0600）提供访问控制，不需要额外的 token 认证

3 双向 Notification

JSON-RPC 2.0 支持双向 notification，用于流式事件推送

4 SSH 隧道远程访问

远程访问通过 SSH 隧道实现，零代码改动

为什么不用 gRPC？（CRI 使用 gRPC）

CRI 选择 gRPC 是因为 protobuf schema 强制校验和代码生成。 ARI 使用 JSON-RPC 是因为更简单，并且下游协议 ACP 也是 JSON-RPC。

跨层保持相同的 wire format 减少了翻译开销

ARI (orchestrator → agentd)  ──JSON-RPC──→  agentd
ACP (agentd → agent-shim)    ──JSON-RPC──→  agent
                                        ↑
                            同协议，零翻译损耗

§3 Workspace 方法

Workspace 方法管理 agent 的工作环境。对标 CRI 的 ImageService。 包含 3 个方法：prepare、list、cleanup。

workspace/prepare

从 Workspace Spec 准备一个 workspace。在 workspace 准备就绪后返回。

// 请求
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "workspace/prepare",
  "params": {
    "spec": {
      "oarVersion": "0.1.0",
      "metadata": { "name": "my-project" },
      "source": { "type": "local", "path": "/home/user/project" }
    }
  }
}

// 响应
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "workspaceId": "ws-abc123",
    "path": "/home/user/project",
    "status": "ready"
  }
}

workspace/list

列出所有 workspace 及其状态。

// 请求
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "workspace/list",
  "params": {}
}

// 响应
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "workspaces": [
      {
        "workspaceId": "ws-abc123",
        "name": "my-project",
        "path": "/home/user/project",
        "status": "ready",
        "refs": ["session-001", "session-002"]
      }
    ]
  }
}

workspace/cleanup

移除 workspace。如果仍有 session 引用此 workspace 则失败。

{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "workspace/cleanup",
  "params": {
    "workspaceId": "ws-abc123"
  }
}

§4 Session 方法

Session 方法管理 agent session。核心方法通过 shim RPC 转发给 agent； ARI 特有的方法处理管理操作。共 9 个方法。

session/new — 创建并启动 agent

创建新 session 并启动 agent 进程。这是最核心的方法。

参数 类型 必填 说明
runtimeClassstring是Agent 类型名称
workspaceIdstring是使用的 workspace
systemPromptstring否Agent 的系统提示
promptstring否初始 prompt
env[]string否额外环境变量
mcpServers[]McpServer否Agent 可用的 MCP 服务
labelsmap否任意标签
roomstring否Room 名称
roomAgentstring否Room 内的 agent 名称
 
// 请求
{
  "jsonrpc": "2.0",
  "id": 10,
  "method": "session/new",
  "params": {
    "runtimeClass": "claude",
    "workspaceId": "ws-abc123",
    "systemPrompt": "你是一个编码 agent。",
    "prompt": "重构 auth 模块，使用 JWT token。",
    "env": ["GITHUB_TOKEN=${GITHUB_TOKEN}"],
    "mcpServers": [
      { "type": "http", "url": "http://localhost:3000/mcp" }
    ],
    "labels": { "task": "auth-refactor" },
    "room": "backend-refactor",
    "roomAgent": "architect"
  }
}

// 响应
{
  "jsonrpc": "2.0",
  "id": 10,
  "result": {
    "sessionId": "session-abc123"
  }
}

内部流程

  1. 1. 查找 runtimeClass 配置（command、args、env）
  2. 2. 生成 OAR config.json（合并参数到 acpAgent 配置）
  3. 3. 写入 config.json 到 bundle 目录
  4. 4. fork/exec agent-shim --bundle <dir>
  5. 5. 返回 session ID

session/prompt — 发送 prompt

向运行中的 session 发送 prompt。agentd 通过 shim RPC 转发给 agent。

{
  "jsonrpc": "2.0",
  "id": 11,
  "method": "session/prompt",
  "params": {
    "sessionId": "session-abc123",
    "prompt": "将 auth 模块重构为使用 JWT token。"
  }
}

响应是 ACP prompt 响应。流式事件作为 JSON-RPC notification 送达。

session/cancel — 取消当前 turn 

{
  "jsonrpc": "2.0",
  "method": "session/cancel",
  "params": {
    "sessionId": "session-abc123"
  }
}

注意：这是 notification（没有 id 字段）。

session/attach / session/detach

Attach 不是 ACP 接管

agentd 始终是 agent 的唯一 ACP client，持有 agent stdio。 Attach 连接只传递 ARI 子集消息。多个连接可以同时 attach（fan-out）。

方向 方法 说明
agentd → clientsession/updateagent 流式输出
agentd → clientsession/stateChange状态变更通知
client → agentdsession/prompt注入 prompt
client → agentdsession/cancel取消当前 turn
 
// Attach
{
  "jsonrpc": "2.0",
  "id": 16,
  "method": "session/attach",
  "params": {
    "sessionId": "session-abc123"
  }
}

// Detach（或直接断开连接，效果相同）
{
  "jsonrpc": "2.0",
  "id": 17,
  "method": "session/detach",
  "params": {
    "sessionId": "session-abc123"
  }
}

其他 Session 方法

session/stop

停止 session，杀死 agent 进程

session/remove

移除 session 元数据（必须已停止）

session/list

列出 session（支持 filter）

session/status

获取 session 详细状态

§5 Room 方法

Room 方法管理 agent 组。对标 CRI 的 PodSandbox 操作。 Room 是协同调度组，支持 agent 间通信。

room/create 

{
  "method": "room/create",
  "params": {
    "name": "backend-refactor",
    "labels": { "project": "auth" },
    "communication": { "mode": "mesh" }
  }
}

room/status 

{
  "result": {
    "name": "backend-refactor",
    "members": [
      { "agentName": "architect", 
        "state": "running" },
      { "agentName": "coder", 
        "state": "paused" }
    ]
  }
}

room/delete 

{
  "method": "room/delete",
  "params": {
    "name": "backend-refactor"
  }
}

删除前所有成员必须已停止

Room 成员状态示例

{
  "members": [
    { "agentName": "architect", "sessionId": "session-abc", "state": "running" },
    { "agentName": "coder", "sessionId": "session-def", "state": "paused:warm" },
    { "agentName": "reviewer", "sessionId": "session-ghi", "state": "running" }
  ],
  "communication": { "mode": "mesh" }
}

列出已注册的 agent 类型（从 Runtime Spec 注册表加载）。

{

    "agents": [
      {
        "name": "claude",
        "annotations": {
          "org.openagents.vendor": "anthropic"

      {
        "name": "gemini",
        "annotations": {
          "org.openagents.vendor": "google"
        },
        "protocol": { "type": "acp", "transport": "stdio" }

  }
}

daemon/status

获取 Daemon 健康状态和运行统计。

{
  "result": {
    "version": "0.1.0",
    "uptime": 3600,
    "sessions": {
      "total": 5,

      "paused": 2,
      "stopped": 1

    "workspaces": { "total": 3, "ready": 3 },

  }
}

§7 事件系统

事件是 agentd 向已连接客户端的推送通知。 使用 JSON-RPC 2.0 notification（没有 id 字段）。

session/update

agent-shim typed event 推送。在 session/prompt 执行期间流式推送。

{
  "jsonrpc": "2.0",
  "method": "session/update",
  "params": {
    "sessionId": "session-abc123",
    "type": "text",
    "content": "我将从分析 auth 模块开始..."
  }
}
type: text type: thinking type: tool_call

session/stateChange

Session 状态转换通知。

{
  "jsonrpc": "2.0",
  "method": "session/stateChange",
  "params": {
    "sessionId": "session-abc123",
    "from": "running",
    "to": "paused:warm",
    "reason": "turn_completed"
  }
}

Session 状态流转

running → paused:warm → paused:cool → stopped

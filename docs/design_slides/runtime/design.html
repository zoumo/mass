§ 架构定位

OCI Runtime Spec 定义了 config.json — 一个可移植、标准化的容器运行时描述。它涵盖了进程、rootfs、挂载、钩子以及平台特定设置（namespaces、cgroups、seccomp）。

OAR Runtime Spec 为 Agent 提供相同的功能：Agent 实例运行时配置的标准化描述。 关键差异在于范围 — Agent 是进程，不是沙箱。我们剥离了所有内核隔离关切，添加了 Agent 原生关切。

OCI

容器运行时

  • • 完整沙箱隔离环境
  • • 内核级安全机制
  • • 文件系统层叠
  • • 网络/存储抽象
OAR

Agent 运行时

  • • 进程级执行单元
  • • 无需内核隔离
  • • 共享工作空间
  • • Agent 协议通信

§ 谁生成谁消费

OCI config.json 不是手写的 — 它由 containerd 生成，由 runc 消费： 

用户编写 Pod Spec (runtimeClassName: kata, image: nginx)
  → kubelet 发送 CRI 请求
  → containerd 解析 runtimeClassName → 找到 handler 配置
  → containerd 从镜像提取 entrypoint, env
  → containerd 生成 config.json (具体进程启动参数)
  → runc 读取 config.json, 启动容器进程

OAR config.json 遵循相同模式： 

orchestrator 发送请求 (runtimeClass: claude, systemPrompt: "...")
  → agentd 解析 runtimeClass → 找到 handler 配置 (command, args, env)
  → agentd 合并 runtimeClass 配置 + 请求参数
  → agentd 生成 config.json (具体 agent 启动参数)
  → agent-shim 读取 config.json, 启动 agent 进程
!

核心原则

Runtime Spec 是一个 具体的、已解析的 运行时描述。 它不包含抽象引用（如 runtimeClass: "claude"）， 而是包含 agent-shim 可直接执行的信息（如 acpAgent.process.command: "npx"）。

这与 OCI config.json 不包含 runtimeClassName 但直接包含 process.args 完全一致。

步骤 OCI (containerd → runc) OAR (agentd → agent-shim)
接收请求 CRI CreateContainer ARI session/new
解析类型 runtimeClassName → handler runtimeClass → handler
生成配置 写入 config.json 到 bundle 写入 config.json 到 bundle
调用运行时 runc create --bundle agent-shim --bundle

§ 为什么是 acpAgent

字段名明确包含 "acp"， 表明 OAR Runtime Spec 仅支持 ACP 协议 Agent。 ACP (Agent Communication Protocol) 是我们支持的唯一 Agent 通信协议标准。

1

通信协议

agent-shim 通过 ACP 协议（stdio JSON-RPC）与 agent 进程通信

2

字段映射

acpAgent.session 字段直接映射到 ACP session/new 参数

3

未来扩展

如需支持非 ACP 协议，需新增顶层字段

ACP 协议栈

上层
agentd
→
协议
ACP (JSON-RPC)
→
下层
agent-shim
stdio 传输 · session/new 初始化 · JSON-RPC 消息格式

§ 两层关切

OCI config.json 将所有信息放在 process 字段。 OAR Runtime Spec 将信息分为两层，对应 agent-shim 消费 config.json 的两个阶段：

Layer 1 — acpAgent.process

如何启动进程

command

"npx"

fork/exec 可执行文件

args

["-y", "@anthropic-ai/claude-code-acp"]

命令行参数

env

["ANTHROPIC_API_KEY=xxx"]

进程环境变量

Layer 2 — systemPrompt + session

ACP session/new 参数

systemPrompt

"You are a backend engineer"

agent 角色约束

mcpServers

[...]

可用 MCP 服务 (acpAgent.session)

⚡ agentRoot — 跨层消费

agentRoot（顶层字段）在两个层都被消费：

Layer 1

agent-shim 解析 agentRoot.path → 规范化绝对路径，设置 cmd.Dir

Layer 2

agent-shim 将解析后的路径作为 cwd 在 ACP session/new 中传递

agent-shim 执行流程

1

读取 acpAgent.process + agentRoot

解析工作空间路径，启动进程（fork/exec，cmd.Dir = 解析后的路径）

2

读取 acpAgent.systemPrompt、解析后的路径、acpAgent.session

发送 ACP session/new

❓ 为什么没有 task/prompt

Runtime Spec 只描述如何创建和初始化 Agent，而非分配什么任务。

Room 场景

所有 Agent 必须先启动并就绪，orchestrator 才通过 ARI session/prompt 发送任务

独立会话

prompt 通过 ARI session/prompt 发送，保持创建与执行分离

§ OCI 对比分析

✓ 从 OCI 保留的字段

OCI 字段 OAR 等价 理由
ociVersion oarVersion Spec 版本控制
process acpAgent 每个工作负载都需要"如何运行"
annotations metadata.annotations 任意 KV 元数据通用有用

✗ 从 OCI 移除的字段

linux.namespaces

Agent 不需要内核级隔离

linux.resources

Agent 不需要 cgroup 约束

linux.seccomp

Agent 不需要 syscall 过滤

mounts

无文件系统层叠，Workspace Manager 准备工作空间

hooks

生命周期钩子在 Workspace Spec 中定义

root

替换为 agentRoot，工作空间在 Room 内共享

+ 为 Agent 新增的字段

acpAgent.systemPrompt

Agent 角色定义和能力约束，由 agentd 从 runtimeClass 配置 + orchestrator 请求填充

acpAgent.process

ACP Agent 进程启动参数，由 agentd 从 runtimeClass 配置填充

acpAgent.session

ACP session/new 参数（mcpServers），指导 agent-shim 如何初始化 ACP 会话

agentRoot 与 Bundle 准备

OCI 的 root.path 是 bundle 内指向容器 rootfs 的相对路径。 OAR 遵循相同模式：

步骤 OCI (containerd + runc) OAR (agentd + agent-shim)
准备工作文件系统 snapshotter 解压镜像层 → rootfs Workspace Manager 克隆 git / 创建 emptyDir
放入 bundle rootfs 写入 bundle 内 root.path symlink 在 agentRoot.path 指向工作空间目录
读取并解析 runc: filepath.Join(bundleDir, root.path) agent-shim: Join + EvalSymlinks
用作工作目录 容器根文件系统 agent 的 cmd.Dir + ACP cwd

共享工作空间模型

OCI rootfs 是每容器独立的（隔离）。OAR 工作空间是共享的 — Room 内多个 Agent 共享同一工作空间目录：

OCI Pod (共享 namespace)

container-1 ─┐
container-2 ─┤── network ns
container-3 ─┘

OAR Room (共享 workspace)

agent-1 bundle/workspace ─┐
agent-2 bundle/workspace ─┤──► /var/lib/agentd/workspaces/ws-abc/
agent-3 bundle/workspace ─┘

§ runtimeClass 在哪

runtimeClass 是 orchestrator → agentd 的参数， 不在 Runtime Spec 中。 agentd 负责将 runtimeClass 解析为 acpAgent 字段中的具体值。

K8s runtimeClass 位置 

Pod Spec → runtimeClassName: kata
         ↓
    (不在 OCI config.json 中)

OAR runtimeClass 位置 

ARI request → runtimeClass: claude
         ↓
    (不在 OAR config.json 中)

runtimeClass 解析流程

containerd 解析

kata → { runtime_type: "io.containerd.kata.v2" }
    → generates config.json: process.args = [...]

agentd 解析

claude → { command: "npx", args: [...], env: {...} }
    → generates config.json: acpAgent.process.command = "npx", ...

详细 runtimeClass 配置参见 agentd 文档

§ config.json 生成流程

Runtime Spec (config.json) 不是手写的。它由 agentd 在创建会话时生成。

1

orchestrator 发送请求

{
  runtimeClass: "claude",
  systemPrompt: "You are a backend engineer",
  env: ["GITHUB_TOKEN=${GITHUB_TOKEN}"],
  mcpServers: [{ type: "http", url: "http://localhost:3000/mcp" }]
}
2

agentd 查找 runtimeClass 配置

claude:
  command: "npx"
  args: ["-y", "@anthropic-ai/claude-code-acp"]
  env:
    ANTHROPIC_API_KEY: "${ANTHROPIC_API_KEY}"
3

agentd 解析工作空间路径

workspaceId: "ws-abc123" → path: "/home/user/project"
4

agentd 生成 config.json

{
  "oarVersion": "0.1.0",
  "metadata": { "name": "session-abc123" },
  "agentRoot": { "path": "workspace" },
  "acpAgent": {
    "systemPrompt": "You are a backend engineer",
    "process": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/claude-code-acp"],
      "env": [
        "ANTHROPIC_API_KEY=sk-ant-xxx",
        "GITHUB_TOKEN=ghp_xxx"
      ]
    },
    "session": {
      "mcpServers": [
        { "type": "http", "url": "http://localhost:3000/mcp" }
      ]
    }
  }
}

← from orchestrator request

← from runtimeClass config

← relative path; agentd symlinks bundle/workspace → ws path

5

agentd 准备 bundle

agentd 准备 bundle: 写入 config.json, symlink bundle/workspace → /var/lib/agentd/workspaces/ws-abc/
agentd 写入 config.json 到 bundle 目录 → agent-shim --bundle

containerd vs agentd 流程对比

步骤 containerd agentd
1. 接收请求 CRI CreateContainer (image + runtimeClassName) ARI session/new (runtimeClass + systemPrompt)
2. 解析类型 runtimeClassName → handler config runtimeClass → handler config
3. 提取运行时信息 image config → process.args, env runtimeClass config → acpAgent.process
4. 合并用户配置 Pod Spec overrides → env, mounts orchestrator request → systemPrompt, env, mcpServers
5. 生成 config 写入 config.json 到 bundle 目录 写入 config.json 到 bundle 目录
6. 调用运行时 runc create --bundle <path> agent-shim --bundle <path>

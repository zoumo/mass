§1 State 模型

Agent 的状态（state）是运行时对 agent 实例的完整描述。状态存储在易失性位置（如 /run/），在主机重启后丢失。持久化元数据由 agentd 负责。

状态属性定义

oarVersion (string, REQUIRED)

OAR Runtime Specification 的版本号

id (string, REQUIRED)

Agent 的唯一标识符，必须在主机上全局唯一

status (string, REQUIRED)

运行时状态：creating | created | running | stopped

pid (int, REQUIRED when created/running)

Agent 进程在主机 OS 中的进程 ID

bundle (string, REQUIRED)

Agent bundle 目录的绝对路径（config.json 所在位置）

annotations (map, OPTIONAL)

与 agent 关联的注解列表

状态示例

{
  "oarVersion": "0.1.0",
  "id": "session-abc123",
  "status": "created",
  "pid": 12345,
  "bundle": "/var/lib/agentd/bundles/session-abc123",
  "annotations": {
    "org.openagents.task": "PROJ-1234"
  }
}

与 OCI State 的对比

OCI state.json OAR state.json 说明
ociVersionoarVersion相同
idid相同
statusstatus相同模式
pidpid相同
bundlebundle相同概念
annotationsannotations相同
namespace_paths—无内核 namespace
cgroup_paths—无 cgroups
设计要点

bundle 字段指向 bundle 目录（config.json 所在位置），而非 state 目录。这镜像了 OCI state 中 bundle 指向 bundle 而非 /run/runc/<id>/ 的设计。

§2 Bundle 格式

Bundle 是一个目录，包含创建 agent 所需的全部信息。运行时从 bundle 目录读取配置。Bundle 的设计与 OCI 的 bundle 概念完全对应。

文件系统布局

/var/lib/agentd/bundles/<agent-id>/   ← bundle dir (bundle field in state.json)
├── config.json
└── workspace -> /var/lib/agentd/workspaces/ws-abc/

/run/agentd/shim/<agent-id>/          ← state dir (ephemeral)
├── state.json
├── agent-shim.sock
└── events.jsonl

Bundle 必需内容

config.json REQUIRED

Agent 配置文件，定义 acpAgent、agentRoot、permissions 等

workspace symlink (agentd 创建)

符号链接，指向 Workspace Manager 准备的工作区目录

agentRoot 与 OCI root 的类比

OCI root.path

bundle 内的相对路径，指向容器的 rootfs。containerd 通过 snapshotter 准备 rootfs。

OAR agentRoot.path

bundle 内的相对路径，指向 workspace。agentd 通过 Workspace Manager 准备 workspace。

关键差异

OCI rootfs 是隔离的（每个容器独立）。OAR workspace 是共享的——Room 中的多个 agent 各有自己的 bundle，但 agentRoot.path 符号链接都指向同一个底层 workspace 目录。

Bundle 生命周期

1 Workspace Manager 准备 workspace 目录（git clone / emptyDir / local）
2 agentd 创建 bundle 目录
3 agentd 写入 config.json，设置 agentRoot.path = "workspace"
4 agentd 创建符号链接：bundle/workspace → <workspace-dir>
5 agentd 调用 runtime 的 create 操作
6 runtime 读取 config.json，解析 agentRoot.path，启动 agent

config.json 结构

{
  "oarVersion": "0.1.0",
  "metadata": { "name": "session-abc123" },
  "agentRoot": { "path": "workspace" },
  "acpAgent": {
    "systemPrompt": "你是后端工程师",
    "process": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/claude-code-acp"],
      "env": ["ANTHROPIC_API_KEY=sk-ant-xxx"]
    },
    "session": {
      "mcpServers": [
        { "type": "http", "url": "http://localhost:3000/mcp" }
      ]
    }
  },
  "permissions": "approve-all"
}
字段解析： agentRoot.path → 相对路径，运行时解析为绝对路径 | acpAgent.process → fork/exec agent 进程 | permissions → fs/terminal 权限策略

§3 生命周期

生命周期描述 agent 从创建到终止的事件时间线。与 OCI 不同，OAR 的 create 操作同时启动进程并完成 ACP 初始化。

生命周期步骤

2 创建运行环境

根据 config.json 配置创建 agent 运行环境。失败时必须生成错误。

3 启动进程

通过 acpAgent.process（command, args, env）使用 fork/exec 启动 agent 进程。

4 ACP initialize 握手

通过 stdio JSON-RPC 完成 ACP initialize 握手。失败时杀死进程，跳至步骤 8。

5 ACP session/new 创建会话

发送 ACP session/new，参数包括 systemPrompt、cwd（解析后的 agentRoot.path）、mcpServers。

6 进入 created 状态

进程运行、ACP 会话建立、agent 就绪等待接收 prompt。

状态转换图

         create
           │
           ▼
      ┌──────────┐
      │ creating  │
      └────┬──────┘
           │ process started + ACP initialized + session established
           ▼
      ┌──────────┐          session/prompt           ┌──────────┐
      │ created   │ ──────────────────────────────► │ running   │
      │ (idle)    │ ◄────────────────────────────── │(prompting)│
      └────┬──────┘          prompt completed        └────┬──────┘
           │                                              │
           │ kill / exit / error                          │ kill / exit / error
           ▼                                              ▼
      ┌──────────┐                                   ┌──────────┐
      │ stopped   │                                   │ stopped   │
      └────┬──────┘                                   └────┬──────┘
           │ delete                                        │ delete
           ▼                                              ▼
       (removed)                                      (removed)

OCI 模式

create 设置环境
但不运行用户程序
用户程序在 start 时启动

OAR 模式

create 启动进程
并完成 ACP 初始化
ACP 是运行时的核心协议

设计决策

如果 ACP 握手失败，agent 实例不被视为成功创建。start 操作当前是 no-op，保留供未来使用，并保持与 OCI create/start 分离的 API 兼容性。

§4 操作语义

运行时必须支持以下操作。这些操作定义的是语义接口，而非命令行 API。

state <agent-id> 查询

返回指定 agent 的完整状态（如 State 节所定义）。

⚠ 如果 agent ID 不存在，必须生成错误。
create <agent-id> <path-to-bundle> 核心

创建新 agent 实例。这是生命周期中最关键的操作。

读取 config.json
解析 agentRoot：filepath.Join + EvalSymlinks
启动进程（fork/exec，cmd.Dir 设为解析路径）
完成 ACP initialize 握手
发送 ACP session/new
写入 state.json
start <agent-id> 保留

当前为 no-op。Agent 在 create 时已启动并完成 ACP 初始化。

💡 保留供未来使用，保持与 OCI API 兼容性。
kill <agent-id> <signal> 终止

向 agent 进程发送信号。默认信号为 SIGTERM。

💡 如果进程在宽限期内未退出，调用方（agentd）负责发送 SIGKILL。
delete <agent-id> 清理

删除 agent，清理 create 步骤创建的资源（如 state 目录）。

⚠ Agent 必须处于 stopped 状态才能删除。删除后 ID 可被后续 agent 使用。

权限策略

策略 行为
approve-all所有操作自动批准
approve-reads只读操作批准，写操作返回 deny
deny-all所有操作返回 deny

§5 事件流

运行时必须产出结构化的 typed event stream，供上层消费。事件持久化到 state dir 下的 events.jsonl，支持 agentd 重连后回放。

事件类型定义

事件类型 ACP 来源 说明
ThinkingEventthought_message_chunkAgent 推理/思考过程
TextEventagent_message_chunkAgent 回复文本片段
ToolCallEventtool_call工具调用开始
ToolResultEventtool_call_update工具调用完成/失败
FileWriteEventfs/write_text_file文件写入操作及结果
FileReadEventfs/read_text_file文件读取操作及结果
CommandEventterminal/*Shell 命令执行及输出
PlanEventplan / plan_update执行计划及状态更新
TurnStartEventprompt 开始标记一个 turn 的开始
TurnEndEventprompt_response标记一个 turn 的结束
ErrorEventACP 错误/进程异常错误信息
持久化要求

Runtime MUST 将 typed events 追加写入 state dir 下的 events.jsonl。这确保 agentd 重连后可以完整回放事件历史，恢复上层状态。

事件流架构

ACP Agent → ACP Events → Runtime (typed) → events.jsonl → agentd
                │                              │
                │                              ├─ ThinkingEvent
                │                              ├─ TextEvent
                │                              ├─ ToolCallEvent / ToolResultEvent
                │                              ├─ FileWriteEvent / FileReadEvent
                │                              ├─ CommandEvent
                │                              ├─ PlanEvent
                │                              ├─ TurnStartEvent / TurnEndEvent
                │                              └─ ErrorEvent

§6 进程分类

运行时启动 agent 进程后，从自身角度看所有 agent 都通过 ACP over stdio 通信。是否经过 wrapper（协议翻译层）对运行时是透明的。

进程架构

# 原生 ACP agent — 直接对接
runtime → ACP stdio → gemini（原生 ACP）

# ACP wrapper — 协议翻译后对接（对 runtime 透明）
runtime → ACP stdio → claude-acp → Claude Code 原生协议
runtime → ACP stdio → pi-acp     → pi RPC → GSD
runtime → ACP stdio → codex-acp  → Codex 原生协议

原生 ACP Agent

直接实现 ACP 协议，无需 wrapper。如 Gemini CLI。

零开销 直接通信

ACP Wrapper

翻译原生协议到 ACP。如 claude-acp、pi-acp、codex-acp。

协议翻译 透明代理
透明性原则

Wrapper 是否存在是 runtimeClass 配置的细节，运行时不感知。所有 agent 从运行时角度看都是相同的 ACP over stdio 接口。

Socket 路径约定

/run/agentd/shim/<session-id>/agent-shim.sock

Socket 与 state.json、events.jsonl 同在 state dir 下。 agentd 重启后扫描 /run/agentd/shim/*/agent-shim.sock 即可重连所有存活的 runtime，无需额外记录 socket 路径。

总结

OCI 兼容设计

State、Bundle、操作语义与 OCI runtime 规范高度对齐，便于集成现有工具链。

ACP 核心协议

Agent Communication Protocol 是运行时的核心，create 时即完成握手和会话建立。

事件驱动架构

Typed event stream 持久化，支持重连回放，构建可靠的 agentd-runtime 通信。
